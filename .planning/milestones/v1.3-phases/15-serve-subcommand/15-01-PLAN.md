```rust
// ---- Phase 15: serve subcommand ------------------------------------------------

/// CLI-01: `mnemonic --help` lists `serve` as a subcommand.
#[test]
fn test_serve_appears_in_help() {
    let bin = binary();
    let output = Command::new(&bin)
        .arg("--help")
        .output()
        .expect("failed to run mnemonic binary");

    let stdout = String::from_utf8_lossy(&output.stdout);
    assert!(
        output.status.success(),
        "--help must exit 0; stderr: {}",
        String::from_utf8_lossy(&output.stderr)
    );
    assert!(
        stdout.contains("serve"),
        "--help must list 'serve' subcommand; got:\n{}",
        stdout
    );
}

/// CLI-01: `mnemonic --help` shows the correct help text for `serve`.
#[test]
fn test_serve_help_text_description() {
    let bin = binary();
    let output = Command::new(&bin)
        .arg("--help")
        .output()
        .expect("failed to run mnemonic binary");

    let stdout = String::from_utf8_lossy(&output.stdout);
    assert!(output.status.success(), "--help must exit 0");
    assert!(
        stdout.contains("Start the HTTP server"),
        "--help must show 'Start the HTTP server' for serve subcommand; got:\n{}",
        stdout
    );
}
```

These tests reuse the existing `binary()` helper function. No `TempDb` needed since `--help` does not touch the database.

The tests verify:
1. `serve` appears in `--help` output (CLI-01 requirement)
2. The help text matches D-03 ("Start the HTTP server")
3. `--help` exits 0

Note: We do NOT test that `mnemonic serve` actually starts the server in a process-based integration test — that would require spawning a long-running process and managing its lifecycle. The existing `tests/integration.rs` already tests the server behavior at the library level, and the dispatch correctness is guaranteed by the Rust type system (exhaustive match). The help-text test proves the subcommand is registered.
