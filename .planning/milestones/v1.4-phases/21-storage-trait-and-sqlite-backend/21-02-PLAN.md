   ```rust
   // Instead of collecting write_ops and doing one big transaction,
   // call write_compaction_result for each cluster (each call is atomic within the backend)
   let mut memories_merged: u32 = 0;
   let mut memories_created: u32 = 0;

   for (new_id, source_ids, merged_content, merged_tags, earliest, merged_embedding) in write_ops {
       match self.backend.write_compaction_result(MergedMemoryRequest {
           new_id,
           agent_id: agent_id.clone(),
           content: merged_content,
           tags: merged_tags,
           embedding_model: self.embedding_model.clone(),
           created_at: earliest,
           source_ids: source_ids.clone(),
           embedding: merged_embedding,
       }).await {
           Ok(_) => {
               memories_created += 1;
               memories_merged += source_ids.len() as u32;
           }
           Err(e) => {
               // Update compact_runs to 'failed'
               let run_id_c = run_id.clone();
               let _ = self.audit_db.call(move |c| {
                   c.execute(
                       "UPDATE compact_runs SET status='failed', completed_at=datetime('now') WHERE id=?1",
                       rusqlite::params![run_id_c],
                   )
               }).await;
               return Err(e);
           }
       }
   }
   ```

   Note: The existing `write_ops` collection loop (lines 301-328) stays mostly the same -- it collects (new_id, source_ids, merged_content, merged_tags, earliest, merged_embedding) per cluster. The change is in how write_ops are consumed: instead of one big db.call transaction, iterate and call `self.backend.write_compaction_result()` per cluster.

   IMPORTANT: For SQLite, each `write_compaction_result` call is a separate transaction. This is acceptable because the current behavior (all clusters in one transaction) is an SQLite-specific optimization. Future backends (Qdrant) will not support multi-cluster transactions anyway. The atomicity guarantee is per-cluster (insert merged + delete sources), which is preserved.

   If preserving the all-clusters-in-one-transaction behavior is critical for SQLite, an alternative is to keep write_ops as-is but that leaks SQLite-specific logic into CompactionService. The per-cluster approach is cleaner for the trait abstraction.
