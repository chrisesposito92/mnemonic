## Phase Details

### Phase 26: Proto Foundation
**Goal**: The gRPC build pipeline is verified and the contract is locked — cargo build succeeds with generated Rust types, incremental builds are fast, and CI passes with protoc installed
**Depends on**: Nothing (first v1.5 phase)
**Requirements**: PROTO-01, PROTO-02, PROTO-03, PROTO-04
**Success Criteria** (what must be TRUE):
  1. `cargo build` succeeds and includes generated gRPC Rust types from mnemonic.proto with all five RPC message shapes present
  2. Running `cargo build` twice in a row completes the second run in under 2 seconds with no build script output (incremental build is not always-dirty)
  3. CI release workflow passes with protoc installed — no cryptic missing-file errors on any build target
  4. `cargo tree -d | grep -E "tonic|prost"` shows zero duplicate entries, and the chosen tonic/prost version is documented in a comment in Cargo.toml
  5. All gRPC dependencies (tonic, prost, tonic-build, tonic-health, tonic-reflection) are behind the `interface-grpc` feature flag — default binary carries no new dependencies
**Plans:** 2/2 plans complete
Plans:
- [x] 26-01-PLAN.md — Proto file, feature-gated Cargo.toml dependencies, and conditional build.rs codegen
- [x] 26-02-PLAN.md — CI release workflow protoc installation

### Phase 27: Dual-Server Skeleton and Auth Layer
**Goal**: `mnemonic serve` starts both REST and gRPC on separate ports simultaneously, the async Tower auth layer rejects invalid tokens and passes open-mode traffic, and grpc_port is configurable
**Depends on**: Phase 26
**Requirements**: SERVER-01, SERVER-02, SERVER-03, AUTH-01, AUTH-02, AUTH-03
**Success Criteria** (what must be TRUE):
  1. `mnemonic serve` starts both servers — a grpcurl health check against the gRPC port returns SERVING, and the REST API continues to respond on its existing port
  2. A gRPC request with an invalid Bearer token returns `Code::Unauthenticated` (not `Code::Internal` or a panic)
  3. A gRPC request with no API keys configured passes through without authentication (open-mode bypass consistent with REST behavior)
  4. Setting `MNEMONIC_GRPC_PORT` changes the gRPC port — both servers start on their respective configured ports
  5. If either port fails to bind (e.g., already in use), the process exits immediately with a clear error rather than partially starting
**Plans:** 2/2 plans complete
Plans:
- [x] 27-01-PLAN.md — Config grpc_port, gRPC module skeleton with tonic-health, dual-port tokio::try_join! startup
- [x] 27-02-PLAN.md — Async Tower auth layer (GrpcAuthLayer) with open-mode bypass, wired into serve_grpc

### Phase 28: Core RPC Handlers, Health, and Discoverability
**Goal**: All four gRPC hot-path operations work end-to-end with correct status codes, scope enforcement, and typed responses — plus standard health reporting and optional grpcurl discoverability
**Depends on**: Phase 27
**Requirements**: GRPC-01, GRPC-02, GRPC-03, GRPC-04, GRPC-05, HEALTH-01, HEALTH-02
**Success Criteria** (what must be TRUE):
  1. A gRPC client can store a memory via StoreMemory and receive back the stored memory with its assigned ID
  2. A gRPC client can issue a SearchMemories RPC and receive ranked results with distance scores — the same memories returned by the equivalent REST search
  3. A scoped API key with agent_id "agent-A" that calls any RPC with agent_id "agent-B" receives `Code::PermissionDenied` — scope enforcement applies to all four handlers
  4. DeleteMemory on a non-existent ID returns `Code::NotFound`; invalid input (empty content, missing required field) returns `Code::InvalidArgument` — not `Code::Internal`
  5. `grpcurl -plaintext localhost:50051 grpc.health.v1.Health/Check` returns SERVING; `grpcurl -plaintext localhost:50051 list` enumerates available services via reflection
**Plans:** 2/2 plans complete
Plans:
- [x] 28-01-PLAN.md — build.rs reflection descriptor, helper functions (api_error_to_status, enforce_scope, memory_to_proto), all 4 RPC handler implementations, tonic-reflection wiring
- [x] 28-02-PLAN.md — Integration tests for all handlers (happy path, error cases, per-handler scope enforcement), health and reflection smoke tests

### Phase 29: StorageBackend Routing Fix
**Goal**: The `mnemonic recall` CLI routes all operations through the StorageBackend trait so that recall and the ListMemories gRPC RPC work correctly regardless of which backend is configured
**Depends on**: Phase 26 (for build stability; can run in parallel with Phases 27-28)
**Requirements**: DEBT-01
**Success Criteria** (what must be TRUE):
  1. `mnemonic recall` with `--storage-provider qdrant` returns results from Qdrant — not from the local SQLite file
  2. `mnemonic recall` with `--storage-provider postgres` returns results from Postgres — confirming the trait routing is not SQLite-specific
  3. All existing `mnemonic recall` tests continue to pass with SQLite as the backend — zero regression from the refactor
**Plans:** 1/1 plans complete
Plans:
- [x] 29-01-PLAN.md — Add init_recall() helper, refactor recall handlers to use StorageBackend trait, add delegation tests
