**Part A: Add a helper function to pre-seed memories in a TempDb**

Since the `remember` subcommand (Phase 17) does not exist yet, tests must insert memory rows directly via rusqlite. Add this helper function after the `TempDb` impl block (after line 57):

```rust
/// Seeds a memory row directly into the SQLite database for testing recall.
/// Uses rusqlite synchronously (not tokio) since test setup is blocking.
fn seed_memory(db_path: &str, id: &str, content: &str, agent_id: &str, session_id: &str, tags: &str, created_at: &str) {
    let conn = rusqlite::Connection::open(db_path).expect("open temp db for seeding");
    // Ensure schema exists (db::open creates it, but we need it before first binary run)
    conn.execute_batch(
        "CREATE TABLE IF NOT EXISTS memories (
            id TEXT PRIMARY KEY,
            content TEXT NOT NULL,
            agent_id TEXT NOT NULL DEFAULT '',
            session_id TEXT NOT NULL DEFAULT '',
            tags TEXT NOT NULL DEFAULT '[]',
            embedding_model TEXT NOT NULL DEFAULT '',
            created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
            updated_at DATETIME
        );
        CREATE TABLE IF NOT EXISTS api_keys (
            id TEXT PRIMARY KEY,
            name TEXT NOT NULL DEFAULT '',
            display_id TEXT NOT NULL,
            key_hash TEXT NOT NULL,
            agent_id TEXT,
            created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
            revoked_at DATETIME
        );"
    ).expect("create schema for seeding");
    conn.execute(
        "INSERT INTO memories (id, content, agent_id, session_id, tags, embedding_model, created_at)
         VALUES (?1, ?2, ?3, ?4, ?5, 'test-model', ?6)",
        rusqlite::params![id, content, agent_id, session_id, tags, created_at],
    ).expect("seed memory row");
}
```

Note: The `api_keys` table must also be created because `db::open` creates it and the binary may reference it. Also note that `vec_memories` virtual table is NOT created here (would require sqlite-vec extension registration) — but recall queries only hit the `memories` table so this is fine.

Actually, since the binary calls `db::open` which creates all tables, and the binary is invoked AFTER seeding, we need to make sure the binary can open a DB that already has a memories table. The binary's `db::open` uses `CREATE TABLE IF NOT EXISTS` so this is safe. But the seed helper should run BEFORE the binary invocation to insert rows. The binary will then open the same DB file and find the pre-existing rows.

However, there's a subtlety: the binary registers sqlite-vec which requires the extension. The seed helper does NOT register sqlite-vec, so it cannot create the `vec_memories` virtual table. The binary will handle that. So the seed helper should only create the `memories` table (and optionally `api_keys` to avoid warnings).

**Simplified approach:** Use the binary itself to initialize the DB schema first (by running a harmless command like `recall` on the empty DB), THEN seed via rusqlite, THEN run the actual test command. OR just seed directly — the binary's `db::open` with `CREATE TABLE IF NOT EXISTS` handles missing tables gracefully, and sqlite-vec registration happens at process startup.

The cleanest approach: seed_memory opens the DB, creates ONLY the memories table with `CREATE TABLE IF NOT EXISTS`, inserts the row, and closes. Then the binary invocation opens the same file, `db::open` adds any missing tables/indexes without conflict.
